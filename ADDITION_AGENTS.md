## ✅ 行为准则

| 准则 | 说明 |
|------|------|
| 确认意图 | 不确定用户意图时，问；先问是不是，再问为什么 |
| 获得许可 | 执行操作前先确定获得了用户许可 |
| 疑问 vs 确定 | 疑问句只讨论，确定句直接执行 |
| 不追问 | 实现后不追问"要提交吗"，等用户主动说 |

---

## ✅ 做事流程

| 步骤 | 说明 |
|------|------|
| 1. 理解需求 | 确认用户要什么，先问是不是再问为什么 |
| 2. 查询与评估 | 搜索资料、分析可行性、准备 2-3 个方案及优缺点 |
| 3. 交互确认 | 与用户讨论并决定最终方案 |
| 4. 实现 | 直接执行；遇到错误先自主排查修复，无法解决时求助 |
| 5. 验证 | 检查结果是否符合预期 |
| 6. 汇报 | 简洁说明完成状态，展示修改对比；验证失败则回滚并告知 |

## 📦 Git 使用策略

| 指令 | 含义 |
|------|------|
| 更新代码仓库 | `git pull`（如未指定仓库，先罗列可更新的仓库并询问） |
| 提交修改记录 | `git commit` |
| 推送修改记录 | `git push` |

| 原则 | 说明 |
|------|------|
| 批量提交 | 累积修改后统一提交，不频繁提交 |
| 提交前确认 | 用 `git status` 和 `git log --oneline -5` 确认 |
| 获得授权 | 提交和推送需获得用户授权 |
| 暂存管理 | 相关修改一起 `git add`，不分开暂存 |
| 回滚机制 | commit 错了用 `git reset --soft HEAD~1` 回退（保留修改，仅撤销提交） |
| 提交规范 | commit message 格式：`类型：描述`（如 `feat: 添加备份功能` / `fix: 修复流程 bug`） |
| 仓库发现 | 目录无 Git 不询问，发现未跟踪仓库时询问是否加入本地笔记（TOOLS.md） |
| 冲突处理 | `git pull` 遇到冲突时，先展示冲突文件列表，让用户决定如何解决 |
| 忽略文件 | 发现未跟踪的敏感文件（.env、密钥、配置文件等）时，提醒添加到 .gitignore |

### 提交类型

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档修改 |
| `style` | 格式调整（不影响代码逻辑） |
| `refactor` | 重构（非新功能、非 bug 修复） |
| `chore` | 构建/工具/配置等杂项 |
